# MySQL 数据类型与范式面试要点

## 一句话总览

MySQL 数据类型不要只背名字，面试重点是：字符串区分定长和变长，`TEXT` 是有字符集语义的大文本，`BLOB` 是二进制字节，时间类型要分清是否受时区影响，金额要用 `DECIMAL/NUMERIC`，范式的核心是减少冗余和异常，尤其要理解联合主键下的“依赖整个主键”。

## 常见数据类型修正版

| 类型 | 类别 | 面试判断 | 说明 |
| --- | --- | --- | --- |
| `CHAR` | 字符串 | 必须掌握 | 定长字符串，最大 255 字符，适合长度固定的数据，例如状态码、固定长度编码。 |
| `VARCHAR` | 字符串 | 必须掌握 | 变长字符串，最大长度受行大小、字符集和其他字段影响，不是固定 65532 字符。 |
| `TEXT` | 大文本 | 必须掌握 | 存储较长文本，受字符集和排序规则影响。 |
| `BLOB` | 二进制 | 必须掌握 | 存储二进制字节，例如图片、音频、文件、加密内容，不受字符集影响。 |
| `DATETIME` | 时间 | 必须掌握 | 保存字面时间，不自动做时区转换，适合业务语义上的固定时间。 |
| `TIMESTAMP` | 时间 | 必须掌握 | 通常按 UTC 存储，读取时按会话时区转换，适合创建时间、更新时间、日志时间。 |
| `BOOLEAN` | 布尔 | 必须纠正 | MySQL 中通常是 `TINYINT(1)` 的别名，不是真正独立的布尔存储类型。 |
| `TINYINT` | 整数 | 必须掌握 | 小整数，1 字节，常用于状态、类型、布尔值。 |
| `BIGINT` | 整数 | 必须掌握 | 大整数，8 字节，适合大 ID、订单号、计数器、雪花 ID。 |
| `DECIMAL/NUMERIC` | 精确小数 | 必须掌握 | 精确十进制，适合金额、财务、费率。不要用 `FLOAT/DOUBLE` 存钱。 |
| `JSON` | 半结构化 | 面试加分 | 存储 JSON 文档，适合扩展属性，但不能滥用来逃避表结构设计。 |

| --- | --- |

## 三大范式怎么说

### 第一范式

字段保持原子性，一个字段里不要塞多个值。

例如 `phone_list = "138xxx,139xxx"` 就不太符合第一范式，因为手机号列表还能继续拆。

### 第二范式

在第一范式基础上，非主键字段必须完全依赖整个主键，不能只依赖联合主键的一部分。

这里的“整个主键”主要是针对联合主键说的。主键不一定只有一个字段，也可以由多个字段共同组成。

例如选课表：

```sql
PRIMARY KEY (student_id, course_id)
```

这时 `(student_id, course_id)` 合起来才表示一条选课记录。

`score` 依赖整个主键，因为必须同时知道“哪个学生”和“哪门课”，才能知道成绩。

但 `student_name` 只依赖 `student_id`，`course_name` 只依赖 `course_id`，它们只依赖联合主键的一部分，所以放在选课表里会违反第二范式。

更合理的设计是：

```text
students
- student_id
- student_name

courses
- course_id
- course_name

student_courses
- student_id
- course_id
- score
```

重点不是“不能用联合主键”，而是用了联合主键以后，表里的非主键字段应该依赖整个联合主键。

### 第三范式

在第二范式基础上，非主键字段不能依赖其他非主键字段，也就是不能有传递依赖。

面试可以这样记：

> 非主键字段依赖主键，依赖整个主键，并且只依赖主键。

其中：

- 依赖主键：对应第一范式之后的基本表设计。
- 依赖整个主键：重点对应第二范式，尤其是联合主键。
- 只依赖主键：重点对应第三范式，避免依赖其他非主键字段。

## `CHAR` 和 `VARCHAR` 的区别

`CHAR` 是定长，`VARCHAR` 是变长。

`CHAR` 适合长度固定的数据。例如性别编码、国家码、固定长度 hash。如果实际内容不足定义长度，存储时会有补齐行为，可能浪费空间。

`VARCHAR` 适合长度变化明显的数据。例如用户名、标题、邮箱、地址。它更节省空间，但需要额外字节记录长度。

不要简单说 `CHAR` 一定快、`VARCHAR` 一定慢。更准确的说法是：

> 固定长度且短的数据可以考虑 `CHAR`，长度变化明显的数据优先考虑 `VARCHAR`。

## `TEXT` 和 `BLOB` 的区别

`TEXT` 存的是文本，数据库会按字符集和排序规则理解它。

常见字符集：

| 字符集 | 说明 |
| --- | --- |
| `utf8mb4` | 最常用，支持中文、英文、emoji，推荐优先使用。 |
| `utf8mb3` | 老版本里的 `utf8`，最多 3 字节，不能完整支持 emoji。 |
| `latin1` | 主要支持西欧字符，不适合中文。 |
| `ascii` | 只适合基础英文字符。 |
| `gbk` | 中文环境里以前常见，现在一般不如 `utf8mb4` 通用。 |

排序规则会影响比较和排序。例如 `utf8mb4_0900_ai_ci` 里的 `ci` 表示大小写不敏感，所以 `A` 和 `a` 比较时可能被认为相等。

`BLOB` 存的是二进制字节。数据库不按字符集解释它，也不关心中文、大小写或文本排序规则。图片、音频、视频、压缩包、加密后的内容更适合用 `BLOB`。

面试可以这样答：

> `TEXT` 是有字符集语义的大文本，适合文章、描述、备注；`BLOB` 是二进制字节，适合文件、图片、音频等。二者都能存大内容，但数据库对它们的理解方式不同。

## `DATETIME` 和 `TIMESTAMP` 的区别

`DATETIME` 保存的是字面时间，不自动做时区转换。

比如写入：

```text
2026-07-10 10:00:00
```

它更像是在记录“业务上就是这个时间”。适合生日、预约时间、业务发生时间等。

`TIMESTAMP` 通常按 UTC 存储，读取时根据当前会话时区转换。它更适合系统自动记录时间，例如创建时间、更新时间、日志时间。

面试可以这样答：

> `DATETIME` 更适合业务语义上的固定时间，`TIMESTAMP` 更适合系统记录时间，因为它会涉及时区转换。

## `TINYINT`、`BIGINT`、`NUMERIC` 的区别

`TINYINT` 是小整数，范围小，占用空间少，常用于状态、类型、布尔值。

例如：

```text
0 = 禁用
1 = 启用
```

`BIGINT` 是大整数，适合 ID、订单号、计数器、雪花 ID 这类可能非常大的整数。

`NUMERIC` 和 `DECIMAL` 通常用于精确小数，尤其是金额。

例如：

```sql
price DECIMAL(10, 2)
```

表示一共 10 位数字，其中小数 2 位。

面试一定要强调：

> 金额用 `DECIMAL/NUMERIC`，不要用 `FLOAT/DOUBLE`，因为浮点数可能有精度误差。

## 面试推荐回答模板

如果被问“`CHAR` 和 `VARCHAR` 怎么选”，可以回答：

> `CHAR` 是定长，适合长度固定且较短的数据；`VARCHAR` 是变长，适合长度变化明显的数据。不能简单说谁一定更快，主要看数据长度是否稳定、空间利用率和业务语义。

如果被问“`TEXT` 和 `BLOB` 有什么区别”，可以回答：

> `TEXT` 存文本，受字符集和排序规则影响；`BLOB` 存二进制字节，不受字符集影响。文章、描述、备注用 `TEXT`，图片、文件、音频、加密内容用 `BLOB`。

如果被问“`DATETIME` 和 `TIMESTAMP` 有什么区别”，可以回答：

> `DATETIME` 保存字面时间，不做时区转换，适合业务固定时间；`TIMESTAMP` 会涉及 UTC 和会话时区转换，适合创建时间、更新时间、日志时间。

如果被问“三大范式怎么理解”，可以回答：

> 第一范式要求字段原子性；第二范式要求非主键字段依赖整个主键，尤其是联合主键时不能只依赖其中一部分；第三范式要求非主键字段只依赖主键，不能再依赖其他非主键字段。核心目的都是减少数据冗余，避免插入、更新、删除异常。

## 记忆口诀

字符串看定长变长，文本看字符集，二进制看字节；时间看时区，金额看精度；范式看依赖，联合主键尤其要看是否依赖整个主键。
