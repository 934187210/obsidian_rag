# Milvus 索引参数与面试要点

## 一句话总览

Milvus 的索引不要只背名字，面试重点是：`dim` 由 embedding 模型决定，`index_type` 决定近似搜索算法，`metric_type` 决定相似度语义，`nlist/nprobe` 只属于 IVF 系列，`M/efConstruction/ef` 属于 HNSW 系列，标量索引用来配合过滤条件。

## 参数表修正版

| 参数 | 位置 | 面试判断 | 说明 |
| --- | --- | --- | --- |
| `dim` | Schema 定义 | 必须掌握 | 向量维度，由 embedding 模型输出决定。插入向量长度必须等于 `dim`。 |
| `emb_len` | 业务代码/封装层 | 不建议当 Milvus 标准参数讲 | Milvus 标准 Schema 参数通常是 `dim`。如果项目里叫 `emb_len`，本质也是 embedding 维度，必须和 `dim` 一致。 |
| `index_type` | 创建索引 | 必须掌握 | 决定索引算法，例如 `FLAT`、`IVF_FLAT`、`IVF_SQ8`、`HNSW`、`DISKANN`。 |
| `metric_type` | 创建索引/搜索 | 必须掌握 | 常见有 `COSINE`、`IP`、`L2`。创建索引和搜索时要保持语义一致。 |
| `nlist` | IVF 创建索引 | 必须纠正 | 表示聚类簇数，和 `dim` 没有必然关系，不是“通常等于 dim”。它应该按数据量、召回率和延迟目标调。 |
| `nprobe` | IVF 搜索 | 必须掌握 | 搜索多少个簇，范围通常不能超过 `nlist`。越大召回越高，延迟也越高。 |
| `limit` | 搜索 | 必须掌握 | 返回 TopK 条结果，例如 `limit=20` 返回最相似的 20 条。 |
| `filter/expr` | 搜索 | 面试加分 | 标量过滤条件，例如租户、类别、时间范围。实际业务检索很常见。 |
| `output_fields` | 搜索 | 面试加分 | 控制返回哪些字段，避免无意义地取回大字段。 |

## 最容易被问错的点

### 1. `dim` 和 `nlist` 不是一回事

`dim=1024` 表示每条向量有 1024 个维度，这是模型输出决定的。

`nlist=1024` 表示 IVF 索引把数据聚成 1024 个簇，这是索引结构决定的。

所以不能说“`nlist` 通常等于 `dim`”。更准确的说法是：`nlist` 要根据数据量、召回率和查询延迟调参。

### 2. `metric_type=IP` 不一定等于余弦相似度

`IP` 是内积。只有当向量已经归一化时，内积排序才和 cosine 排序等价。

面试可以这样答：

> 如果 embedding 已经做了归一化，`IP` 可以近似/等价用于 cosine 检索；如果没有归一化，`IP` 会受到向量模长影响，和余弦相似度不是一回事。

### 3. `nprobe` 是召回和延迟的旋钮

`nprobe` 越大，搜索的簇越多，越接近全量搜索，召回通常更高，但查询更慢。

面试可以这样答：

> IVF 的核心调参是 `nlist` 和 `nprobe`。`nlist` 决定建索引时分多少簇，`nprobe` 决定查询时扫多少簇。线上一般通过压测在召回率和 P95/P99 延迟之间取平衡。

## 常见索引怎么回答

### FLAT

精确暴力搜索，不做近似。召回最高，但数据量大时慢。适合小数据集、离线评估、做召回基线。

### IVF_FLAT / IVF_SQ8 / IVF_PQ

IVF 的思路是先聚类分桶，搜索时只查部分桶。

`IVF_FLAT` 不压缩向量，精度相对好。

`IVF_SQ8` 用标量量化压缩向量，节省内存，但有精度损失。

`IVF_PQ` 用乘积量化进一步压缩，更省内存，但精度损失也可能更明显。

面试重点不是背全名，而是说清楚：IVF 用 `nlist/nprobe` 控制召回和延迟，SQ/PQ 是用压缩换内存和速度。

### HNSW

图索引，工程里很常见，适合低延迟、高召回的 ANN 检索。

关键参数：

| 参数 | 位置 | 说明 |
| --- | --- | --- |
| `M` | 建索引 | 每个节点连接多少邻居，越大图越密，召回更好但内存更高。 |
| `efConstruction` | 建索引 | 建图时搜索范围，越大索引质量更好但构建更慢。 |
| `ef` | 搜索 | 查询时搜索范围，越大召回更高但延迟更高。 |

面试可以这样答：

> HNSW 通过多层近邻图做启发式搜索，不需要全量扫描，所以查询快。它通常比 IVF 更容易获得较高召回，但内存占用也会更明显。

### DISKANN

适合数据规模很大、内存放不下时，用 SSD/磁盘能力支撑向量检索。重点是省内存、支撑大规模，但部署和磁盘性能更关键。

### SPARSE_INVERTED_INDEX

用于稀疏向量，比如 BM25、关键词权重、混合检索里的 sparse retrieval。

RAG 面试里可以这样说：

> 稠密向量负责语义相似，稀疏向量负责关键词匹配。实际 RAG 可以 dense + sparse 混合召回，再做 rerank。

### 标量索引

标量索引用来加速过滤条件，不是向量相似度本身。

常见类型包括：`INVERTED`、`BITMAP`、`Trie`、`STL_SORT`。

面试要知道业务查询经常是：先按 `tenant_id/category/time` 过滤，再做向量 TopK。因此标量索引和向量索引经常一起用。

## 面试推荐回答模板

如果被问“Milvus 索引怎么选”，可以回答：

> 我会先看数据规模、延迟目标、召回要求和内存预算。小数据或需要精确结果时用 `FLAT` 做基线；线上低延迟高召回常考虑 `HNSW`；数据量较大、需要控制内存和吞吐时考虑 `IVF_FLAT`、`IVF_SQ8` 或 `IVF_PQ`；数据大到内存放不下时再看 `DISKANN`。如果是 RAG 场景，我还会考虑 dense + sparse 混合检索，并配合标量索引做租户、类别、时间过滤。

如果被问“`nlist` 和 `nprobe` 怎么理解”，可以回答：

> `nlist` 是建索引时分多少个簇，`nprobe` 是查询时搜索多少个簇。`nprobe` 越大，召回越高但延迟越高。`nlist` 和向量维度 `dim` 没有必然关系，需要通过数据规模和压测调参。

如果被问“`IP`、`COSINE`、`L2` 怎么选”，可以回答：

> 这取决于 embedding 模型和业务相似度定义。很多语义向量检索会用 cosine；如果向量已归一化，也可以用 `IP` 得到等价排序；`L2` 更像欧氏距离。关键是建索引和搜索时 metric 语义要一致，插入和查询向量的预处理也要一致。

## 记忆口诀

`dim` 看模型，`metric` 看语义，`index_type` 看规模和延迟，IVF 调 `nlist/nprobe`，HNSW 调 `M/ef`，线上一定结合过滤条件和压测结果。
