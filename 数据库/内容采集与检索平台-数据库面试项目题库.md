# 内容采集与检索平台：数据库项目面试题库

> 本文是一套面试演练场景，不代表真实项目经历。回答中的规模、延迟和保留周期应替换为面试者能够证明的真实数据。

## 一分钟项目介绍

这是一个内容采集与检索平台。系统定时从多个来源采集文章，完成正文清洗、规范化、去重、分块和向量化，最终支持关键词和语义检索。

我会按数据职责拆分存储：MySQL 保存来源、文章元数据、处理任务和状态；Redis 保存热点结果、短期幂等标记和任务协调信息；对象存储保存原始正文和冷日志；Milvus 保存向量及检索所需的轻量元数据。这样做的原因是不同数据的访问模式不同：业务状态需要事务，向量需要近似检索，原始内容需要低成本持久化，日志需要按生命周期归档。

面试回答的主线是：先明确数据和一致性要求，再选择存储结构、事务边界、索引、缓存和扩展方式，最后说明每个方案承担的代价。

## 一、架构与数据建模

### 1. 为什么不把所有数据都放在 MySQL？

**推荐回答：**

MySQL 负责关系型业务数据最合适，例如来源、文章、任务状态、重试次数和处理结果。这些数据需要条件查询、唯一约束和事务更新。原始正文通常体积大、修改少，放对象存储更便宜；向量检索需要 ANN 索引，放 Milvus 更符合访问模式；日志则需要按时间保留和清理。分层不是为了堆组件，而是让每类数据使用匹配它的存储能力。

**追问边界：**

如果数据量和查询复杂度仍然很小，单独引入对象存储、Redis 或向量库反而会增加运维成本，应先用简单方案验证需求。

### 2. MySQL 中会设计哪些核心表？

**推荐回答：**

可以先设计以下几类表：

- `source`：来源站点、抓取规则、启用状态。
- `article`：文章业务主键、来源、规范化 URL、标题、摘要、内容版本和生命周期状态。
- `ingest_task`：一次采集或处理任务，包括任务类型、状态、重试次数、错误原因和时间字段。
- `article_chunk`：文章分块编号、文本摘要、向量 ID 和处理状态。
- `outbox_event`：需要异步投递给向量化或索引服务的事件。

文章和任务是不同生命周期的对象，不能只在文章表里放一个 `status` 就覆盖所有处理阶段。任务表用于记录尝试和失败，文章表用于表达当前业务状态。

**追问边界：**

表结构应从查询和状态转移反推，而不是先把所有字段都放进一张“万能表”。

### 3. 文章如何去重？

**推荐回答：**

先定义业务上的唯一身份。常见做法是对规范化后的来源标识和 URL 建唯一键，同时保留内容哈希用于检测 URL 不变但正文变化的情况。写入时使用唯一约束兜底，再配合幂等插入或更新逻辑。Redis 可以减少重复请求，但不能作为最终唯一性保证，因为它可能过期、丢失或与 MySQL 状态不一致。

**追问边界：**

URL 规范化必须处理协议、默认端口、尾部斜杠、追踪参数和重定向，否则同一篇文章可能产生多个业务身份。

### 4. 为什么文章状态和任务状态要分开？

**推荐回答：**

文章状态表示业务对象当前是否可用，例如 `discovered`、`cleaned`、`published`、`archived`；任务状态表示某次处理尝试是否成功，例如 `pending`、`running`、`succeeded`、`failed`。一个文章可能经历多次清洗或向量化重试，因此两者不能混为一个字段。状态变更应通过明确的状态机限制非法跳转，并记录版本或更新时间，便于并发控制和排查。

**追问边界：**

状态机不应该只写在文档里；关键状态迁移应在服务层校验，并用条件更新防止旧任务覆盖新状态。

### 5. 主键为什么倾向使用自增 ID 或有序 ID？

**推荐回答：**

InnoDB 的主键通常是聚簇索引，行数据存放在主键索引叶子节点。短且大体有序的主键能减少二级索引体积和随机页写入，因此内部表可以使用自增 ID 或有序分布式 ID。对外暴露的文章标识可以另设不可枚举的业务 ID，避免把内部主键直接当作安全边界。

**追问边界：**

自增 ID 不承诺连续；回滚、并发和批量插入都可能产生间隙。是否使用 UUID 还要结合索引大小、排序需求和跨库生成能力评估。

### 6. 是否使用物理外键？

**推荐回答：**

如果数据都在同一个 MySQL 实例、数据关系稳定且完整性由数据库兜底很重要，可以使用外键。若未来要拆库、跨服务迁移或高并发批量写入，通常会减少物理外键，改用应用校验、事件顺序、定时巡检和修复任务维持逻辑关系。无论选哪种方式，都要明确删除策略和孤儿数据处理方案。

**追问边界：**

“不用外键”不是“不做一致性校验”；它只是把一部分约束从数据库层移到了应用和数据治理层。

## 二、事务、幂等与并发控制

### 7. 采集任务重复投递时，如何保证幂等？

**推荐回答：**

我会把幂等拆成三层：数据库唯一键保证同一业务身份不能重复创建；任务表记录任务类型和当前版本，避免同一阶段被重复执行；外部副作用通过 `outbox_event` 或幂等消费键控制。Redis 锁可以减少同一时间的重复执行，但最终结果仍必须由数据库约束和条件更新兜底。

**追问边界：**

重试必须区分可重试错误和不可重试错误，并设置退避、最大次数和人工处理状态，不能无限重试。

### 8. 一次采集任务的事务边界怎么划？

**推荐回答：**

事务只包住必须原子完成的本地数据库操作，例如创建任务、更新文章版本和写入待处理事件。网络请求、正文下载、向量化等耗时操作不放进同一个长事务，否则会长时间持锁。事务提交后由异步消费者处理下一阶段；如果需要保证“业务写入”和“事件投递”不丢失，可以使用 Outbox 模式，再由发布器可靠投递。

**追问边界：**

Outbox 解决的是本地数据库写入与事件记录的一致性，不能自动保证外部向量库操作成功，后者仍需要重试、补偿和状态对账。

### 9. 任务状态并发更新如何避免旧任务覆盖新任务？

**推荐回答：**

可以给任务增加 `version` 或 `updated_at` 条件，使用类似下面的条件更新：

```sql
UPDATE ingest_task
SET status = 'succeeded', version = version + 1
WHERE id = ? AND version = ? AND status = 'running';
```

受影响行数为 0 时，说明任务已经被其他执行者更新，当前结果不能直接覆盖。乐观锁适合冲突不高、任务执行时间较长的场景；如果竞争激烈且临界区很短，才考虑更明确的悲观锁或队列串行化。

**追问边界：**

版本号主要防止丢失更新，不会自动解决业务层面的重复副作用。

### 10. 遇到死锁怎么排查？

**推荐回答：**

先通过数据库死锁日志或监控拿到参与事务、SQL、索引和锁等待信息，再检查是否存在相反的加锁顺序、范围扫描过大、缺少索引或事务持有时间过长。修复方向包括统一访问顺序、缩小事务、补充合适索引和减少批量锁范围。应用层仍要把死锁视为可重试错误，但重试必须有次数上限和退避，不能只靠重试掩盖问题。

**追问边界：**

行锁本质上锁索引记录；SQL 没有有效利用索引时，锁定范围可能扩大，不能只看“代码里操作的是一行”。

### 11. 这个场景选 RR 还是 RC？

**推荐回答：**

文章状态和任务状态通常需要清晰的并发可见性。InnoDB 默认 RR 可以让同一事务内的普通快照读保持稳定；如果某些批处理更关注并发、能接受每次读取看到新提交数据，可以评估 RC。选择前要验证锁等待、死锁、复制配置和读一致性要求，不能用“查询就 RC、核心就 RR”作为固定规则。

**追问边界：**

要区分普通快照读和 `SELECT ... FOR UPDATE` 等当前读；范围当前读可能使用间隙锁或临键锁。

### 12. MySQL 和 Milvus 的写入如何保持一致？

**推荐回答：**

不把它们包装成一个跨系统事务，而是采用最终一致的状态机：MySQL 先记录文章分块和待向量化事件，向量化服务成功写入 Milvus 后，再回写向量 ID 和完成状态。失败时保留可重试状态；定时对账任务检查 MySQL 中已完成但 Milvus 缺失、或 Milvus 已存在但 MySQL 未确认的记录，并执行补偿或人工处理。

**追问边界：**

向量库的幂等键应由文章版本和分块编号组成，不能只用随机生成的向量 ID，否则重试会产生重复向量。

## 三、索引、慢 SQL 与分页

### 13. 文章列表应该建什么索引？

**推荐回答：**

先收集实际查询，例如“某来源下按发布时间倒序查已发布文章”和“查某个任务状态的待处理任务”，再按过滤、排序和选择性设计联合索引。索引列顺序不是固定模板，需要结合等值条件、范围条件、排序方式和数据分布验证。对低选择性字段单独建索引未必有收益，最终要用执行计划和线上指标判断。

**追问边界：**

索引能减少扫描，但会增加写入维护和缓存占用；每个索引都要对应一类稳定的查询路径。

### 14. `(source_id, status, published_at)` 联合索引怎么理解？

**推荐回答：**

如果主要查询是固定 `source_id`、固定 `status`，再按 `published_at` 排序或做范围过滤，这个顺序有机会同时支持过滤和排序。但如果查询模式经常变化，或者 `status` 选择性很低，实际收益可能不同。不能仅凭“字段都在 SQL 中出现”决定索引顺序，要结合最左前缀和 `EXPLAIN` 验证。

**追问边界：**

包含关系不等于可以无条件删除短索引；短索引可能更小，缓存命中和写入成本也可能更好。

### 15. 深分页为什么慢？如何改？

**推荐回答：**

`LIMIT offset, size` 需要先扫描并丢弃前面的记录，offset 越大成本越高。文章列表可以改为基于稳定排序键的游标分页，例如按 `(published_at, id)` 组成严格顺序，下一页使用上一页最后一条记录作为边界。这样要保证排序键稳定、组合条件有对应索引，并处理相同时间戳和删除数据的情况。

**追问边界：**

游标分页更适合“继续向后翻”，不天然支持跳到任意页；产品需求和数据库性能之间需要取舍。

### 16. 慢 SQL 的排查流程是什么？

**推荐回答：**

先从慢查询日志、APM 或数据库监控确认真实 SQL、参数、频率和 P95/P99 延迟，再用 `EXPLAIN` 或 `EXPLAIN ANALYZE` 检查访问类型、候选索引、实际行数和排序临时表。然后判断问题属于缺索引、条件选择性差、回表过多、深分页、锁等待还是数据模型不合适。优化后要用相同数据分布和代表性流量复测，不能只看单次执行时间。

**追问边界：**

“加索引”不是默认答案；索引可能让写入更慢，也可能因为优化器估算错误而没有被使用。

### 17. 软删除字段会带来什么问题？

**推荐回答：**

软删除要求所有业务查询正确过滤有效状态，并重新检查唯一键、索引选择性和归档策略。可以把 `deleted_at` 或状态字段纳入主要查询路径，但是否有效仍要用执行计划和实际数据分布验证。历史数据持续增长后，应考虑按生命周期归档、分区清理或物理删除；对于需要彻底删除的数据，软删除不满足要求。

**追问边界：**

软删除是数据生命周期方案的一部分，不是单独加一个 `is_deleted` 字段就完成了。

## 四、缓存、读写与扩展

### 18. 检索结果如何使用缓存？

**推荐回答：**

可以采用 Cache Aside：先查 Redis，未命中再查数据库或检索服务，成功后写入缓存；更新时先更新主数据，再删除或失效缓存。缓存键应包含租户、查询参数、过滤条件和版本信息，避免不同权限或不同版本的数据混用。缓存只用于降低重复读取成本，不能作为唯一数据源。

**追问边界：**

缓存更新存在短暂不一致，需要结合业务可接受的陈旧时间；强一致场景应直接读取权威存储或采用版本校验。

### 19. 如何处理缓存击穿和热点 key？

**推荐回答：**

对热点 key 可以使用短时互斥、请求合并、随机过期时间或预热，避免大量请求同时回源。对不存在的数据也可以短时间缓存空值，降低穿透。治理时要限制单个缓存值大小、设置合理 TTL，并监控命中率、回源量和 Redis 延迟。

**追问边界：**

分布式锁只能减少并发回源，不能替代数据库唯一约束，也要处理锁超时、持有者崩溃和续期问题。

### 20. 什么时候考虑读写分离？

**推荐回答：**

当主库的读压力已经影响写入延迟，且读请求能够接受可量化的复制延迟时，才评估读写分离。普通文章浏览和历史列表可以读副本；刚发布后必须立即可见、任务状态确认等请求需要走主库或采用一致性路由。上线前要验证复制延迟、故障切换、连接池、读请求路由和副本读到旧数据时的产品表现。

**追问边界：**

读写分离不能解决所有性能问题；如果瓶颈是慢 SQL、锁竞争或磁盘 IO，增加副本可能只是把问题扩散。

### 21. 什么时候考虑分区、分表或分库？

**推荐回答：**

先区分问题是数据生命周期、单表访问性能、单实例容量，还是团队边界和故障隔离。按时间归档和清理可以考虑分区或冷热表；单表索引和写入无法满足目标时才评估水平分表；单实例容量、连接和故障域成为瓶颈时才考虑分库。每次拆分都要提前设计路由键、全局 ID、跨分片查询、备份恢复和数据迁移方案。

**追问边界：**

固定行数不是通用拆分阈值，真实触发条件应来自压测、监控和增长预测。

### 22. 为什么不把向量直接存 MySQL？

**推荐回答：**

MySQL 可以保存向量的版本、模型、维度和 Milvus ID 等元数据，但高维向量近似检索需要专门的索引和搜索参数。把向量原始值放在 MySQL 会增加行宽和存储压力，也不能自然提供 Milvus 的 ANN 能力。两边通过稳定业务键关联，并用异步补偿处理最终一致性。

**追问边界：**

如果数据量很小、只做离线验证，直接存 MySQL 或文件可能更简单；是否引入向量数据库应由规模和检索目标决定。

## 五、日志、归档与恢复

### 23. 业务数据和原始日志如何分开？

**推荐回答：**

文章状态、任务状态和错误摘要属于业务数据，需要事务、查询和权限控制；采集原始响应、调试日志和链路日志属于可观测性或归档数据，应该按生命周期保存。MySQL 可以保存错误摘要和关联 ID，原始日志写入对象存储或日志系统，避免用业务表长期承载大体积追加数据。

**追问边界：**

日志保留策略应区分热检索、离线分析和长期归档，不能只设置一个无限期保留周期。

### 24. redo、undo、binlog 分别做什么？

**推荐回答：**

`undo log` 保存旧版本，用于回滚和 MVCC；`redo log` 记录存储引擎的修改信息，用于崩溃恢复；`binlog` 记录服务层的逻辑变更，用于复制、审计或基于日志的恢复。它们的职责不同，不能用“都有日志所以一样”来回答。提交路径还需要协调 redo 和 binlog，避免恢复和复制看到不一致的事务结果。

**追问边界：**

普通快照读依赖 undo 版本链，但当前读和写操作仍然会使用锁；不要把 MVCC 说成所有读都无锁。

### 25. 采集失败如何恢复？

**推荐回答：**

失败任务要记录可重试原因、尝试次数、下次执行时间和最后错误摘要。网络超时、限流和临时存储不可用通常可以退避重试；解析错误、权限错误和内容不符合规则可能需要人工处理。任务状态、重试次数和下一次执行时间应在数据库中持久化，避免进程重启后丢失调度意图。

**追问边界：**

重试要配合幂等键，否则每次重试都可能重复创建文章、写入向量或发送通知。

### 26. 如何保证原始内容可追溯？

**推荐回答：**

原始正文写入对象存储后，在 MySQL 保存对象键、内容哈希、采集时间、来源和版本号。规范化正文和分块结果可以重新生成，但原始版本用于审计、回放和解析规则升级。对象存储数据也要有保留、校验、备份和访问权限策略，不能只保存一个 URL。

**追问边界：**

内容版本变化时，文章业务状态、分块版本和向量版本必须可关联，否则重新索引时可能混用新旧内容。

## 六、Trade-off 综合追问

### 27. 这个项目最重要的数据库设计取舍是什么？

**推荐回答：**

最重要的取舍是把“业务权威状态”和“异步派生数据”分开。MySQL 是文章和任务状态的权威来源，Redis、对象存储和 Milvus 都是针对访问模式的派生或专用存储。这样可以接受短暂的最终一致性，换取更合适的检索能力、成本和扩展性；代价是必须建设事件重试、对账、补偿和监控。

**追问边界：**

如果业务要求写入后立即完成语义检索，就需要重新评估同步链路、用户体验和一致性成本，不能继续假设所有处理都异步。

### 28. 你如何证明一次优化真的有效？

**推荐回答：**

先明确目标指标，例如查询 P95 延迟、锁等待时间、缓存命中率、任务成功率、复制延迟或归档成本，再建立优化前基线。修改索引、事务或缓存后，用代表性数据和流量复测，并检查副作用，例如写入变慢、缓存不一致、空间膨胀或故障恢复变复杂。没有指标和对照实验的“优化”只能算猜测。

**追问边界：**

单次 SQL 变快不代表系统变好；还要看高并发、长尾延迟、写放大、故障恢复和运维成本。

## 快速复述模板

### 被问“为什么这样设计？”

> 我先按数据的访问模式和一致性要求拆分职责：事务型元数据放 MySQL，原始大对象放对象存储，向量检索放 Milvus，热点结果放 Redis。这样做牺牲了跨系统强一致性，但通过 Outbox、幂等键、重试和对账保持最终一致，同时降低单一数据库承载所有工作负载的压力。

### 被问“怎么保证数据不重复？”

> Redis 只做快速拦截，MySQL 唯一键才是最终约束；任务使用幂等键和状态版本，外部副作用通过 Outbox 和幂等消费处理。重试时不重新生成业务身份，并通过对账任务发现跨系统状态不一致。

### 被问“怎么优化慢查询？”

> 先从监控确认真实 SQL、参数、频率和长尾延迟，再用执行计划判断扫描、排序、回表、锁等待或分页问题。索引顺序根据查询路径设计，修改后用代表性数据复测，同时观察写入成本和缓存占用。

### 被问“什么时候拆库分表？”

> 不按固定行数直接拆分，而是根据容量、延迟、写入压力、备份窗口和增长趋势判断。先优化索引、归档、分区和读写路径；确认单实例已经成为瓶颈后，再设计路由键、全局 ID、跨库查询、迁移和恢复方案。
