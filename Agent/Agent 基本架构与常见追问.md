# Agent 的基本架构与常见追问

> 来源：小红书问答整理
>
> 核心结论：完整的 Agent 是一套以 LLM 为决策中枢、能够自主逼近目标的闭环系统，而不是“调用一次大模型”。

## 一、四个不可替代的核心组件

### 1. LLM：理解与决策中枢

LLM 接收用户输入、工具返回结果以及记忆召回内容，负责判断下一步行动：继续思考、调用哪个工具、调整计划，还是直接给出最终答案。

没有 LLM，工具、记忆和规划都只是被动的基础设施。

### 2. 工具（Tools）：连接外部世界

LLM 本身是语言处理器，不能直接联网、读写文件、执行代码或访问业务系统。搜索、代码执行器、数据库、业务 API 等工具，构成 Agent 与外部世界交互的入口。

工具调用遵循“决策与执行分离”：

1. 工具定义只向模型说明 `name`、`description` 和 `parameters`。
2. LLM 根据说明决定是否调用、调用哪个工具以及参数值，并返回结构化调用请求。
3. 业务程序真正执行工具。
4. 执行结果回填上下文，由 LLM 再次决策。

模型负责“决定做什么”，程序负责“真正执行”。

### 3. 记忆（Memory）：保持状态，避免失忆

记忆通常分为两层：

| 层级 | 主要载体 | 作用 | 特点 |
| --- | --- | --- | --- |
| 短期记忆 | Context Window、会话状态 | 保存当前任务的输入、中间结果和工具轨迹 | 容量有限，通常随任务结束清理 |
| 长期记忆 | 向量数据库或其他持久化存储 | 保存跨任务、跨时间仍有价值的事实与经验 | 容量较大，需要主动检索和召回 |

短期记忆承载当前工作状态，长期记忆承载可复用知识。两者都不能简单等同于“把所有历史消息永久塞进上下文”。

### 4. 规划（Planning）：拆解复杂目标

规划决定 Agent 能否处理复杂任务。常见有两种形态：

- **先规划后执行**：先输出完整计划，再按步骤执行。
- **边执行边规划**：每完成一步，根据工具结果动态调整后续步骤。

前者结构清晰、便于审查；后者适应性更强，适合结果不确定的任务。

## 二、核心运行闭环

典型数据流如下：

```text
目标
  -> 规划（plan）
  -> LLM 决策（结合当前步骤、短期记忆、长期记忆召回）
  -> 工具执行（tools.execute）
  -> 结果写入短期记忆
  -> LLM 再次决策
  -> 循环执行，直到形成 final_answer
```

在这个闭环中：

- LLM 始终是决策者；
- 工具是执行者；
- 记忆负责保存状态和提供历史信息；
- 规划负责把目标转化为可执行步骤。

可以类比一家公司：LLM 是老板，工具是执行团队，记忆是档案室，规划是项目经理。四者各司其职，才构成可自主运行的 Agent。

## 三、框架的定位

LangChain、LlamaIndex、AutoGen 等框架，本质上都在围绕 LLM、Tools、Memory、Planning 这几类能力做封装，只是侧重点和抽象粒度不同。

在生产系统中，Agent 也不一定要让 LLM 控制整个流程。更稳妥的实践通常是从确定性的 Workflow 起步，只在确实需要判断和适应性的节点引入 LLM 决策。

## 四、常见追问一：MCP 对工具系统带来了什么改变？

### 结论

MCP（Model Context Protocol）把工具接入从“每个 Agent 或框架单独写适配器”，推进为标准化、可发现、可插拔的客户端/服务端协议。

### 具体改变

1. **工具描述统一**：工具可以用统一方式声明名称、用途、参数和能力，客户端不必为每个工具重新约定格式。
2. **接入方式可复用**：同一个 MCP Server 可以被不同的 Agent、宿主应用或模型客户端复用，降低重复集成成本。
3. **能力按需发现**：Agent 可以先了解服务端提供的工具或资源，再决定调用什么，不必把每个业务系统的细节硬编码到 Agent 中。
4. **工具系统更易插拔**：替换搜索、数据库、代码执行器等外部能力时，Agent 的核心决策逻辑可以保持相对稳定。
5. **治理边界更清晰**：协议负责能力暴露与调用通信；服务端仍可以在自己的边界内实现鉴权、审计、限流和执行安全策略。

### 不会改变的职责边界

MCP 解决的是工具接口和连接方式的标准化，并不会替代 Agent 的规划和决策：

- LLM 仍负责判断是否调用工具、调用哪个工具以及传入什么参数；
- MCP Server 或业务程序仍负责实际执行；
- 权限控制、危险操作审批、结果可信度校验和失败重试，仍需要由应用或工具服务设计；
- MCP 也不会自动提升 LLM 的规划质量。

因此，MCP 可以理解为“工具系统的统一插座和通信约定”，而不是一个自动运行 Agent 的大脑。

## 五、常见追问二：多 Agent 的记忆如何设计？

### 结论

不建议在“所有 Agent 共享一个全局记忆”和“每个 Agent 完全私有”之间二选一。更稳妥的方案是：**私有工作记忆为主，受治理的共享记忆为辅，按需检索和传递**。

### 推荐的分层设计

| 记忆层 | 典型内容 | 访问方式 |
| --- | --- | --- |
| Agent 私有短期记忆 | 当前子任务、中间推理、工具调用轨迹 | 仅当前 Agent 使用 |
| 共享任务状态 | 总目标、任务分工、进度、阻塞项、已确认产物 | 协作 Agent 按权限读写 |
| 共享长期知识 | 经验证的事实、文档摘要、代码结论、业务规则 | 通过语义检索按需召回 |
| 领域私有知识 | 某个专业 Agent 负责维护的领域资料 | 领域 Agent 主写，其他 Agent 按权限读取 |

### 设计原则

1. **共享事实，不共享全部思维过程**：共享任务状态、结论和产物即可，不必广播每个 Agent 的完整推理草稿。
2. **按需召回，不全量注入**：只把与当前子任务相关的记忆放入上下文，减少噪声和上下文污染。
3. **带来源和版本**：重要记忆应记录来源、时间、负责人或版本，便于验证和回滚。
4. **明确写入权限**：例如由协调 Agent 更新项目看板，领域 Agent 维护专业知识，避免多个 Agent 无约束地覆盖同一事实。
5. **处理冲突和过期**：对矛盾结论设置优先级、人工确认或重新检索机制，不能因为某条记忆被召回就默认它正确。
6. **共享最小必要信息**：跨 Agent 传递任务所需的摘要、结构化结果和引用，避免泄露不必要的上下文。

### 一个直观类比

每个 Agent 都有自己的工作笔记；项目看板记录团队共同的任务状态；正式文档和知识库保存经过确认、可长期复用的事实。团队共享的是“可协作的事实”，而不是每个人脑中的全部内容。

## 六、最终记忆点

Agent 的基本架构可以压缩为一句话：

> **LLM 负责决策，Tools 负责执行，Memory 负责保持状态，Planning 负责拆解目标；它们通过循环协作，让系统逐步逼近最终结果。**

在此基础上，MCP 主要标准化工具的接入与复用；多 Agent 记忆则应采用分层、按需共享、带治理的设计。
