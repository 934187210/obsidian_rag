# Caddy、SQLite、MySQL 基础知识

标签：#技术知识 #Web服务器 #数据库 #部署 #选型

## 核心结论

Caddy、SQLite、MySQL 解决的是不同层面的问题。

- Caddy 是 Web 服务器和反向代理，主要负责把外部请求安全地转发到应用服务，并自动处理 HTTPS。
- SQLite 是嵌入式关系型数据库，通常就是一个本地 `.db` 文件，适合轻量、单机、低运维场景。
- MySQL 是独立运行的数据库服务器，适合多用户、高并发、长期生产业务系统。

## Caddy 是干嘛用的

Caddy 类似 Nginx，但配置更简单，尤其擅长自动 HTTPS。

常见用途：

- 反向代理：把 `https://api.example.com` 转发到后端服务，例如 `127.0.0.1:3000`。
- 自动 HTTPS：自动申请、续期 TLS 证书，并可自动把 HTTP 跳转到 HTTPS。
- 静态文件服务：托管前端打包产物，例如 React/Vue 的 `dist`。
- 统一入口网关：多个后端服务时，由 Caddy 统一接收外部请求，再转发到不同服务。

典型配置：

```caddyfile
example.com {
    reverse_proxy 127.0.0.1:3000
}
```

这表示用户访问 `https://example.com` 时，Caddy 负责 HTTPS 和入口流量，然后把请求转给本机 `3000` 端口的应用。

一句话：Caddy 是用来把应用服务安全、稳定、方便地暴露到公网的入口服务器。

## SQLite 和 MySQL 的区别

一句话区别：SQLite 是一个本地数据库文件，MySQL 是一个数据库服务器。

| 对比项 | SQLite | MySQL |
| --- | --- | --- |
| 运行方式 | 嵌入到应用里，不需要单独启动服务 | 需要启动 MySQL Server |
| 数据存储 | 通常是一个 `.sqlite` 或 `.db` 文件 | 由 MySQL 服务管理数据目录 |
| 部署成本 | 很低，拷贝文件即可使用 | 需要安装、配置、账号、权限、端口 |
| 并发能力 | 适合轻量读写，写入并发较弱 | 适合多用户、高并发读写 |
| 网络访问 | 默认本地文件访问 | 天然支持远程连接 |
| 权限管理 | 主要依赖文件权限 | 有用户、密码、角色、授权体系 |
| 适合场景 | 本地应用、小工具、移动端、测试、单机服务 | Web 后端、业务系统、多用户平台、生产数据库 |
| 运维复杂度 | 很低 | 更高，但能力更完整 |

## 选型建议

优先选 SQLite 的场景：

- 桌面软件、CLI 工具、本地缓存。
- 单机小服务。
- 原型开发和测试环境。
- 不想维护数据库服务，数据量和并发压力都不高。

优先选 MySQL 的场景：

- 多人同时访问的后台系统。
- 电商、SaaS、管理平台、线上 API 服务。
- 需要账号权限、备份、监控、远程连接。
- 长期生产业务，需要更强的并发和运维能力。

SQLite 不是玩具数据库，它很稳定、可靠，只是目标不是大规模并发服务器，而是轻量、嵌入式、零运维。

## 和项目部署的关系

一个常见 Web 项目里三者可能这样分工：

```text
用户浏览器
  -> Caddy
  -> 后端应用服务
  -> SQLite 或 MySQL
```

- Caddy 负责入口流量、域名、HTTPS、反向代理。
- 后端应用负责业务逻辑。
- SQLite 或 MySQL 负责存储业务数据。

如果只是本地开发或轻量单机服务，可以是 Caddy + 后端 + SQLite。

如果是正式生产、多用户、高并发系统，更常见是 Caddy + 后端 + MySQL。
